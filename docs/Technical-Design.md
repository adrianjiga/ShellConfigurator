# Technical Design

## TOML Generator Pipeline

**File**: `src/generators/starship.ts`

The generator takes a `WizardState` and produces a complete `starship.toml` string. The pipeline has three stages.

### Stage 1: Format String Construction

`buildFormatString(modules)` converts a module list into a Starship format string:

```
['directory', 'git_branch', 'git_status'] → '$directory$git_branch$git_status'
```

The `character` module is filtered out here — it gets special placement after the line break.

### Stage 2: Layout Assembly

`generateToml()` assembles the final `format` value using `$fill` for right-alignment:

```
$directory$git_branch$git_status  $fill  $cmd_duration$time  \n  $character
├── left modules ──────────────┘  │      ├── right modules ─┘   └── prompt line
                                  │
                            fills remaining space
```

This produces a two-line prompt:

```
Line 1: [left info]                          [right info]
Line 2: ❯ _
```

**Why `$fill` instead of `right_format`**: Starship's `right_format` maps to the shell's native RPROMPT, which renders on the cursor line (line 2). This causes right-side modules to appear on a different line than the info modules. `$fill` keeps everything in a single `format` string, avoiding shell-dependent RPROMPT behavior.

### Stage 3: Module Block Generation

`moduleBlock(id, state)` produces the TOML `[section]` for each module. Blocks adapt to two state values:

- **`hasNerdFont`**: switches between icon symbols and text fallbacks
- **`colorScheme`**: applies color styles from `COLOR_STYLES` to directory, git_branch, and git_status

Example output for `git_branch` with Nerd Font and default colors:

```toml
[git_branch]
symbol = " "
style  = "bold purple"
```

### Dollar Sign Escaping

The dollar character symbol requires two layers of escaping:

1. **Starship level**: `$` is the variable interpolation prefix. A literal `$` must be `\$`.
2. **TOML level**: `\$` is not a valid escape in double-quoted TOML strings. The value must use TOML literal strings (single quotes) so the backslash passes through.

Result in generated TOML:

```toml
success_symbol = '[\$](green)'
```

### Fill Block Generation

When `rightModules` is non-empty, the generator emits a `[fill]` block with an invisible space symbol:

```toml
[fill]
symbol = " "
```

This prevents Starship's default fill behavior (visible dots on light terminals) while keeping left and right modules properly separated.

### Final Output Structure

```toml
# Generated by ShellConfigurator
# https://github.com/shellconfigurator

format       = "$directory$git_branch$git_status$fill$cmd_duration\n$character"
add_newline  = true

[fill]
symbol = " "

[character]
success_symbol = '[❯](green)'
error_symbol   = '[❯](red)'

[directory]
style             = "bold blue"
truncation_length = 3
truncate_to_repo  = true

[git_branch]
symbol = "on "
style  = "bold purple"

...
```

## Shell RC Generator

**File**: `src/generators/shellRc.ts`

### Config File Writing

`writeStarshipConfig(toml)` writes the generated TOML to `~/.config/starship.toml`:

1. Ensures `~/.config/` exists via `mkdirSync({ recursive: true })`
2. Overwrites the file with `writeFileSync()`

### Shell Init Line Injection

`applyShellConfig(shellId)` appends the Starship init command to the shell's RC file:

```
~/.zshrc         → eval "$(starship init zsh)"
~/.bashrc        → eval "$(starship init bash)"
~/.config/fish/config.fish → starship init fish | source
```

**Pipeline**:

1. Look up shell definition via `getShell(id)`
2. If `rcFile` is `null` (nushell, powershell): return manual instructions
3. Create parent directory if missing (important for Fish: `~/.config/fish/`)
4. Read existing file content
5. **Idempotency check**: if the init line is already present, skip
6. Append with a comment header:
   ```
   # Added by ShellConfigurator
   eval "$(starship init zsh)"
   ```

The idempotency check uses a simple `string.includes()` — it only matches the exact init line, so a commented-out line (`# eval "$(starship init bash)"`) does not prevent re-addition.

## Installer Task System

**Files**: `src/screens/InstallingScreen.tsx` + `src/services/installer.ts`

### Task Construction

`buildTaskList(state)` creates the task queue from wizard state:

| Task        | Condition                                                       | Action                             |
| ----------- | --------------------------------------------------------------- | ---------------------------------- |
| Starship    | Always                                                          | Install if not already present     |
| Nerd Font   | `nerdFontToInstall` is set and is not the sentinel `__select__` | Download and install font          |
| Shell(s)    | Selected shell not in `installedShells`                         | Install via package manager        |
| Set default | `setDefaultShell` is set                                        | Run `chsh -s <path>`               |
| Config      | Always                                                          | Generate and write `starship.toml` |
| RC files    | Always                                                          | Append init lines to shell configs |

### Task Lifecycle

Each task transitions through these statuses:

```
pending → running → done
                  → failed (with error message)
                  → skipped (e.g., Starship already installed)
```

Status is displayed with icons:

```
[✓] Starship                     (done, green)
[~] Nerd Font (JetBrains Mono)   (running, yellow)
[ ] Write starship.toml          (pending, gray)
[✗] Install fish                 (failed, red)
    Error: apt-get not found       (error detail)
[–] Set zsh as default shell     (skipped, gray)
```

### Execution Flow

Tasks run sequentially in a `useEffect` with a `useRef` guard to prevent double-execution during React strict mode / dev re-renders:

```typescript
const ran = useRef(false);
useEffect(() => {
  if (ran.current) return;
  ran.current = true;
  let cancelled = false;
  // ... run tasks ...
  return () => {
    cancelled = true;
  };
}, []);
```

Each task is wrapped in try-catch. Failures are recorded but do **not** halt the pipeline — remaining tasks still execute.

After all tasks complete, a 1200ms delay lets the user see the final state before auto-advancing to the Done screen. The final task list is passed to DoneScreen via `installResults` in `WizardState`, so DoneScreen can display actual success/failure status for each task.

### Command Execution

`installer.ts` uses `spawnSync` with `stdio: 'inherit'` for all shell commands. This means:

- `sudo` password prompts appear directly in the terminal
- Command output streams to stdout/stderr in real time
- The Ink UI pauses during execution (Ink handles this gracefully)

The `runCommand` helper checks three failure modes in order: spawn errors (`result.error`), signal kills (`result.signal`), and non-zero exit codes (`result.status`). This produces clear error messages in all cases, including when a child process is killed by a signal (where `status` is `null`).

### Platform-Aware Font Installation

`installNerdFont` uses platform detection for the fonts directory:

- **macOS** (`process.platform === 'darwin'`): `~/Library/Fonts` — macOS auto-detects fonts here, so `fc-cache` is skipped
- **Linux**: `~/.local/share/fonts` — requires `fc-cache -fv` after extraction

## Detection System

**File**: `src/services/detector.ts`

### Package Manager Detection

`detectPackageManager()` runs a prioritized detection chain:

```
1. which brew       → 'brew'      (macOS / Linuxbrew)
2. which pacman     → 'pacman'    (Arch-based)
3. /etc/os-release  → ID match:
   ├── ubuntu, debian, linuxmint, pop, elementary → 'apt'
   ├── fedora, rhel, centos, rocky, alma          → 'dnf'
   └── arch, manjaro, endeavouros, cachyos, garuda → 'pacman'
4. which apt-get    → 'apt'       (fallback binary check)
5. which dnf        → 'dnf'       (fallback binary check)
6. fallback         → 'script'    (curl install script)
```

**Why command checks come first**: Homebrew can be installed on Linux, and `which brew` is faster than reading `/etc/os-release`. Pacman is checked early because Arch-based distros don't always have a predictable OS ID.

**Why binary fallback exists (steps 4-5)**: Some minimal containers or custom distros don't have `/etc/os-release` but do have `apt-get` or `dnf` in PATH.

### OS Release Parsing

`readOsReleaseId()` reads `/etc/os-release` and extracts the `ID` field:

```
ID=ubuntu     → 'ubuntu'
ID="rocky"    → 'rocky'
ID='arch'     → 'arch'
```

Handles both quoted and unquoted values.

### Shell Detection

`detectInstalledShells()` checks for each shell binary via `which`:

| Shell ID   | Binary Checked |
| ---------- | -------------- |
| bash       | `bash`         |
| zsh        | `zsh`          |
| fish       | `fish`         |
| nushell    | `nu`           |
| powershell | `pwsh`         |

Returns an array of `ShellId` values for shells found in PATH.

### Starship Detection

`isStarshipInstalled()` runs `starship --version`:

- Success: returns `{ installed: true, version: "starship 1.x.x" }`
- Failure: returns `{ installed: false }`

Used by both WelcomeScreen (to show status) and InstallingScreen (to skip installation).

## Error Handling Strategy

The app handles errors at different levels:

| Layer                            | Strategy                                                             |
| -------------------------------- | -------------------------------------------------------------------- |
| Detection (`detector.ts`)        | Swallow errors, return safe defaults (`false`, `null`, empty arrays) |
| Installation (`installer.ts`)    | Throw on failure — caught by InstallingScreen                        |
| Task runner (`InstallingScreen`) | Catch per-task, record error, continue remaining tasks               |
| Config writing (`shellRc.ts`)    | Throw on failure — caught by InstallingScreen                        |

This means a font download failure won't prevent config file writing, and a missing `chsh` binary won't block RC file updates.
